# Repo Orientation: ccsessions

> Generated by /orient:orient. Re-run to update.

## One-line purpose
`ccsessions` is a Go terminal UI for browsing Claude Code session history for the current working directory, with search, metadata, and transcript inspection.

## Primary language(s)
Go

## Pipeline / workflow stages
1. `cmd/ccsessions/main.go` starts the Bubble Tea program and hands control to the UI model.
2. `internal/ui/model.go` initializes search, list, and detail panes and reacts to keyboard and resize events.
3. `internal/claude/sessions.go` maps the current working directory to `~/.claude/projects/<sanitized-path>` and loads matching `.jsonl` session files.
4. The parser normalizes raw Claude log records into semantic transcript entries such as prompts, assistant text, tool calls, tool results, progress rows, and metadata events.
5. The UI filters sessions by summary and search text, then renders the selected session with scan-friendly styling in the detail pane.

## Core concepts
**Project-scoped session discovery**: The app does not browse all Claude history globally. It derives one project-history directory from the current working directory and only loads `.jsonl` files from there. This logic lives in `internal/claude/sessions.go`.

**Semantic transcript normalization**: Raw Claude records are richer than simple chat messages. The parser classifies records into prompt, assistant text, tool call, tool result, progress, thinking marker, and metadata entries in `internal/claude/sessions.go`.

**Bubble Tea model-driven UI**: The interface is a single stateful model that owns search state, filtered sessions, selection, focus, and viewport rendering. That interaction loop is centered in `internal/ui/model.go`.

**Scan-oriented transcript rendering**: The UI is designed for visual scanning rather than raw log dumping. Prompt indentation, tool/result blocks, and muted progress/meta rows are all expressed in `internal/ui/model.go`.

## Common gotchas
**The app is cwd-sensitive**: If the viewer shows no sessions, the problem may be the current working directory rather than the parser. `projectHistoryDir` in `internal/claude/sessions.go` only checks one sanitized path under `~/.claude/projects`.

**Role is not enough to interpret a log line**: Tool results often arrive in top-level `type:"user"` records, so styling by `role` alone will misclassify them. Use the normalized entry kinds in `internal/claude/sessions.go` instead.

**Tests are structurally expected but currently sparse**: CI in `.github/workflows/build.yml` runs `go test ./...`, but the repo currently has little or no dedicated test coverage. Changes to parsing and rendering need careful manual validation.
